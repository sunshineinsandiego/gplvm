# Multimodal Shared Gaussian Process Latent Variable Model (Shared GPLVM)

Welcome to the **Shared GPLVM** repository. This project uses machine learning to find hidden connections between two very different types of data:
1.  **Medical Imaging:** High-dimensional ultrasound videos of patellar tendons.
2.  **Tabular Data:** Clinical health scores, biomechanics, and athlete game statistics.

Our goal is to learn a "Shared Latent Space"—a hidden map of patient health—that explains both the anatomy seen in the scan and the performance seen on the court.

---

## 1. The Intuition: Plato's Cave for Medicine

Imagine you are a doctor trying to understand the "True Health" of an athlete's knee. You cannot see "True Health" directly; it is an abstract concept. However, you can observe two things that result from it:
1.  **The Shadow on the Wall (Image):** You can take an ultrasound scan.
2.  **The Report Card (Data):** You can measure their pain levels and jump height.

A **Shared GPLVM** assumes that both the Image ($Y_1$) and the Data ($Y_2$) are generated by the same underlying, hidden variable ($X$).

If we can learn this hidden $X$, we can do magic tricks:
*   **Imputation:** If a patient has an ultrasound but forgot to fill out their pain survey, we can find their $X$ using the image, and then use $X$ to predict what their pain score *should* be.
*   **Discovery:** We can see if "structural damage" in the image actually correlates with "poor performance" in the stats, or if they are disconnected.

---

## 2. The Math (Simplified)

We model the relationship as:

$$ Y_{img} = f_1(X) + \text{noise} $$
$$ Y_{tab} = f_2(X) + \text{noise} $$

*   **$X$ (Latent Space):** A low-dimensional vector (e.g., 12 numbers) representing the patient's state. We assume $X$ follows a standard Bell Curve (Gaussian Prior).
*   **$f_1$ and $f_2$ (The Mappings):** These are functions that translate the hidden state $X$ into the observed data.
    *   $f_1$ translates "Health" $\to$ "Image Features".
    *   $f_2$ translates "Health" $\to$ "Clinical Scores".

Because the relationship between anatomy and pain is complex and non-linear, we use **Gaussian Processes (GPs)** to model $f_1$ and $f_2$. GPs are great because they don't just give a prediction; they give a prediction *plus uncertainty*.

---

## 3. Design Choices: Why did we build it this way?

### Q: Why "Two GPs" if it's a "Shared" model?
You might notice in the code (`src/shared_gplvm.py`) that we initialize `self.gp_img` and `self.gp_tab`.

**"I thought we wanted a shared model, not two different models?"**

We **do** have a shared model because they share the **Input ($X$)**.
However, we need two separate **Translators (GPs)** because the **Outputs ($Y$)** are completely different languages.

*   **GP 1 ($f_1$):** Needs to map $X$ to 768 dimensions of visual data (smooth, continuous pixel features).
*   **GP 2 ($f_2$):** Needs to map $X$ to 17 dimensions of clinical data (integers, scores, different scales).

If we used a single GP, we would be forcing the model to use the exact same mathematical "smoothness" (kernel lengthscale) to describe both pixel intensity and pain scores, which doesn't make sense. By splitting them, we allow the model to learn: *"The image changes slowly as health gets worse, but pain scores jump up suddenly."*

They remain "Shared" because during training, we force them to agree on a single $X$ for every patient.

### Q: Why Variational Inference?
Standard GPs scale poorly ($\mathcal{O}(N^3)$). If we have 10,000 video frames, a standard GP would crash the computer.
**Variational Inference** allows us to approximate the GP using a small set of "Inducing Points" (e.g., 50 anchor points). This makes the model fast and scalable to large datasets.

Crucially, Variational Inference allows us to handle **Missing Data**. If a patient is missing their Tabular data ($Y_2$), we don't have to throw them away. We just rely on the Image ($Y_1$) and the Prior ($X \sim \mathcal{N}(0,1)$) to figure out where they sit in the latent space.

## 4. Codebase Walkthrough

### 1. `src/compile_dataset.py` (The Data Engineer)
This script prepares the data. It:
*   Crawls the `scan/` directory for ultrasounds (single images, videos, DICOM multiframes, or DICOM sequences).
*   Reads `PPTTUS.csv` (clinical) and `game_stats.csv` (basketball).
*   **Frame Extraction & Selection:** By default, it detects and extracts **ALL** available frames from the sequence using `--n_keyframes all`. If a subset is desired (e.g., to save processing time on massive video files), you can pass an integer (like `--n_keyframes 10`). This triggers a Determinantal Point Process (DPP) algorithm to pick the most visually diverse subset of frames.
*   **Output:** It creates `data/manifest.csv`, a master list linking every extracted image frame to its corresponding patient stats. Each row contains the specific `frame_id`, path to the raw file, and the full suite of static and dynamic clinical variables for that frame.

### 2. `src/shared_gplvm.py` (The Brain)
This file defines the model architecture.
*   **`SharedVariationalGPLVM`**: The container class. It holds the learnable Latent Embeddings ($X$) and the two GP sub-models.
*   **`LatentGP`**: A standard Variational GP module. We instantiate two of these: one for images, one for tabular data.

### 3. `src/train_gplvm.py` (The Teacher)
This script trains the model.
*   **`MultimodalDataset`**: A smart dataloader. It reads the CSV and loads images. Crucially, it runs the **DINOv3** vision model *on-the-fly* to convert images into feature vectors (embeddings).
*   **The Loop:** It optimizes the "ELBO" (Evidence Lower Bound). It tries to move the latent points $X$ around until they maximize the probability of observing the data we actually see.

### 4. `src/impute_and_evaluate.py` (The Test)
This script tests if the model actually learned anything useful.
*   **Reconstruction:** Can we recreate the data from the learned $X$?
*   **Inference Mode (`image_only`):** This is the "Magic Trick". It takes a new image, forgets everything else, and tries to find the best $X$ using *only* that image. Then, it projects that $X$ to predict the missing clinical scores.
*   **Attribution:** It calculates correlations to tell us: *"Latent Dimension 3 is highly correlated with Tendon Thickness."*

## 1. Background: What is a Shared GPLVM? (Undergraduate Level)
## 1. The Intuition: Plato's Cave for Medicine

Imagine you have two different "views" of a patient:
1. **View 1 (The Image):** A highly detailed, complex ultrasound scan of their knee.
2. **View 2 (The Spreadsheet):** A neat row of numbers containing their clinical health (like a tendon thickness measurement) and their basketball performance (points, rebounds).
Imagine you are a doctor trying to understand the "True Health" of an athlete's knee. You cannot see "True Health" directly; it is an abstract concept. However, you can observe two things that result from it:
1.  **The Shadow on the Wall (Image):** You can take an ultrasound scan ($Y_{img}$).
2.  **The Report Card (Data):** You can measure their pain levels and jump height ($Y_{tab}$).

It is very hard to directly draw a line showing how pixel #402 in an ultrasound affects a player's free-throw percentage. Instead of trying to connect them directly, a **Shared Gaussian Process Latent Variable Model (Shared GPLVM)** takes a different approach.
A **Shared GPLVM** assumes that both the Image and the Data are generated by the same underlying, hidden variable ($X$). If we can learn this hidden $X$, we can perform:
*   **Imputation:** If a patient has an ultrasound but missing clinical data, we find their $X$ using the image, then predict the missing scores.
*   **Discovery:** We can visualize if "structural damage" in the image actually correlates with "poor performance" in the stats.

It assumes that both the complex image and the neat spreadsheet are products of a hidden, underlying "state" specific to that patient—a low-dimensional **Latent Space**. 
Think of this latent space as the "true" overall physical health of the player's knee. We can't measure this "true health" directly. But we know that this hidden state casts two shadows:
- It generates the structural shapes we see in the **ultrasound**.
- It generates the pain scores and performance metrics we record in the **spreadsheet**.
---

Because we use **Gaussian Processes (GPs)**—a statistical tool that excels at modeling smooth, non-linear relationships with built-in uncertainty—we can mathematically reverse-engineer this hidden state! 
## 2. Methodology & Technical Details

Even better, if a player is missing their spreadsheet data (maybe they didn't fill out the clinical survey), the model uses their ultrasound image to find their location in the hidden "true health" space. Once we know where they are in that hidden space, the model can reliably predict what their survey answers *would have been*, along with a confidence interval (uncertainty) telling us how sure it is about the prediction.
### The Math (Simplified)
We model the relationship as two independent Gaussian Processes sharing a latent input:
$$ Y_{img} = f_1(X) + \epsilon_1 $$
$$ Y_{tab} = f_2(X) + \epsilon_2 $$

----
*   **$X$ (Latent Space):** A low-dimensional vector (e.g., 12 dims) representing the patient's state, assumed to follow a standard Normal Prior $\mathcal{N}(0, I)$.
*   **$f_1, f_2$ (The Mappings):** Non-linear functions modeled by GPs. $f_1$ maps to 768-dim image embeddings; $f_2$ maps to clinical scores.

## 2. Specific Considerations for this Repository
### Architecture: Variational Inference
Standard GPs scale cubically ($\mathcal{O}(N^3)$). With video data ($N \approx 30,000$ frames), this is infeasible.
We use **Variational Inference** with Inducing Points ($M \approx 50$).
*   **Complexity:** Reduces to $\mathcal{O}(NM^2)$, linear with respect to frames.
*   **Missing Data:** The Variational Prior allows us to handle missing tabular data naturally. If $Y_{tab}$ is missing, the model relies on $Y_{img}$ and the Prior to position the patient in $X$.

Our dataset presents a unique set of challenges and opportunities:
### Key Algorithms
### Key Algorithms & Mathematical Intuition

1.  **Determinantal Point Process (DPP):** Used in `compile_dataset.py`.
    Raw ultrasound sequences contain huge frame counts ($N \approx 150-300$ individual images per 5 seconds). Passing 150 nearly identical images per patient into our GPLVM bloats the dataset with redundant data. We need to extract the 10 most *structurally diverse* frames that summarize the entire sweep. We do this using a **Determinantal Point Process (DPP)**.
    
    **Here is the underlying mathematics taught conceptually for undergraduates:**
    *   **Step 1: Embedding (The Fingerprint).** First, every frame in the video is passed through a Vision Transformer (DINOv3). This converts the image of the knee into a 768-dimensional vector (an array of 768 numbers).
    *   **Step 2: The Similarity Kernel ($K$).** We stack all 150 vectors into a matrix $X$. We then measure how similar *every* frame is to *every other* frame by computing the dot product:
        $$ K = X X^T $$
        This creates a $150 \times 150$ grid (the Kernel Matrix). If Frame 1 is completely identical to Frame 2, their intersection in grid $K$ will be $1.0$. If they show completely different angles of the tendon, it will be closer to $0$.
    *   **Step 3: Geometry & Volume.** In linear algebra, if you take a subset of vectors and calculate their *Determinant*, that number calculates the geometric "volume" of the space they create. If you pick 3 frames that look very similar, their vectors point in the same direction, and the "volume" they span collapses to zero. If you pick 3 completely different frames, they point in orthogonal directions, forming a wide, 3D shape with maximum volume.
    *   **Step 4: The Selection Metrics.** Our DPP algorithm evaluates combinations of frames and chooses the subset of 10 frames that maximizes this volume. We track this using two metrics:
        *   **`diversity_logdet`:** The logarithm of the determinant. Our algorithm runs a greedy loop, constantly picking the next frame that increases this `diversity_logdet` the most.\n            *   **What do the numbers mean? (-26 vs. 0)** Because our embedding vectors are L2-normalized (scaled so their lengths equal 1), the maximum possible determinant for a perfectly orthogonal (100% unique) set of frames is mathematically capped at $1.0$. The natural logarithm of $1.0$ is $0$. Therefore, **`diversity_logdet` will virtually always be negative** (bounded $\\le 0$). A score of `+26` is mathematically impossible for an L2-normalized correlation matrix. A score like `-26` is completely normal for a 5-second ultrasound sweep—it just means the 10 frames inherently share a lot of overlapping physical structure, which naturally pushes the overall \"volume\" of the subset very close to zero.\n        *   **`coverage`:** A check on our work. This calculates the average similarity between the 10 frames we picked and the original 150 frames. A high coverage ensures that no unique structure shown in the video was accidentally left out of our 10-frame summary!
2.  **NaN Masking:** We implement a custom Marginal Log-Likelihood that masks out `NaN`s in the tabular data, allowing training on partial patient records.
3.  **Z-Score Normalization:** All continuous inputs/outputs are normalized to mean 0, std 1 to ensure stability of the RBF kernels.

*   **Low Patient Count ($N$):** We currently have around 46 to 92 ultrasound images. Deep learning models (like VAEs or multi-modal autoencoders) generally fail with such a small $N$ because they overfit. GPs are uniquely suited for low-$N$ regimes because they gracefully fall back on their priors and maintain robust uncertainty bounds.
*   **High-Dimensional Imaging ($D_1$):** Raw ultrasound videos are impossibly huge. We solve this by passing the scans through frozen Vision Foundation Models (**DINOv3** and **MedSAM2**). This reduces the images to dense feature embeddings (e.g., 768 or 1024 dimensions). While smaller than raw pixels, this is still high-dimensional, making a dimensionality-reduction technique like GPLVM ideal.
*   **Low-Dimensional Tabular Data ($D_2$):** Features extracted from `PPTTUS.csv` and `game_stats_CU.csv` range from categorical (Position) to continuous (Tendon Thickness, VISA score, Points Per Game). These features must be carefully normalized so they contribute appropriately to the joint marginal likelihood alongside the high-dimensional image embeddings.
*   **Missingness:** In our clinical dataset, missing values will occur naturally (e.g., athletes missing surveys or non-players missing game stats). The Shared GPLVM handles this naturally by "anchoring" the patient in the latent space using their available modality.
### Prediction Strategy
*   **Training:** Regression (Gaussian Likelihood) to preserve ordinality.
*   **Evaluation:** Classification (Buckets) for clinical utility.
    *   **V_SCORE:** `<70` (Pathological), `70-80`, `80-90`, `90+` (Healthy).
    *   **AT_THICK:** `<4.5mm` (Healthy), `4.5-5.5mm` (Thickened), `>5.5mm` (Pathological).

## 3. Implementation Options and Considerations
## 3. Codebase Structure

When building a Shared GPLVM, we have several architectural and framework choices:

### Architecture Options
1.  **Exact Shared GPLVM:** Inverts the full $N \times N$ covariance matrix.
    *   *Pros:* Mathematically exact point estimations, no approximations.
    *   *Cons:* Computes distances between *every single data point*, scaling cubically at $\mathcal{O}(N^3)$.
    *   *Verdict for us:* If we only had 92 still images ($N=92$), an Exact GPLVM would be viable. However, to capture the full structural variance of the tendon, we treat *every video frame* independently. This causes $N$ to skyrocket into tens of thousands of frames. A standard GPLVM is computationally impossible in this regime because inverting a $30,000 \times 30,000$ matrix every training step would exhaust memory and compute limits.
2.  **Variational Shared GPLVM (vGPLVM):** Introduces a small set of "inducing points" (e.g., 50-100 anchor points) to approximate the posterior distribution.
    *   *Pros:* Drops the computational complexity to $\mathcal{O}(NM^2)$, making it easily scalable to tens of thousands of video frames on a GPU. Furthermore, instead of optimizing exact points, it places a **Gaussian probability distribution** (Bayesian prior) over the latent space $X$. This mathematical prior gives us a rigorous, built-in mechanism to handle missing clinical tabular data by falling back on the image likelihood to anchor the probability coordinates.
    *   *Cons:* Slightly more complex to train and tune bounds (ELBO).

### Framework Options
1.  **GPy / paramz:** The historic gold standard for GPLVMs in Python.
    *   *Downside:* Built on NumPy. It cannot seamlessly backpropagate gradients into PyTorch models or leverage GPU acceleration effectively alongside neural networks.
2.  **Pyro / NumPyro:** Highly flexible Probabilistic Programming Languages.
    *   *Downside:* Steep learning curve to encode custom cross-modal likelihoods manually.
3.  **GPyTorch:** A highly optimized GP library built directly on PyTorch.
    *   *Upside:* Excellent GPU support, seamlessly integrates with PyTorch-based frozen encoders (DINOv3/MedSAM2), and has natively implemented `gpytorch.models.gplvm` modules that we can extend for Shared/Multimodal setups.
*   **`src/compile_dataset.py`**: The Data Engineer. Crawls `scan/` directories, merges with `PPTTUS.csv` and `game_stats.csv`, runs DPP keyframe selection, and outputs `data/manifest.csv`.
*   **`src/shared_gplvm.py`**: The Model. Defines `SharedVariationalGPLVM` using GPyTorch. Contains two `LatentGP` modules sharing a learnable `X`.
*   **`src/train_gplvm.py`**: The Trainer. Implements the training loop, optimizing the ELBO. Handles on-the-fly DINOv3 feature extraction, Patient-ID splitting, and NaN masking.
*   **`src/impute_and_evaluate.py`**: The Evaluator. Performs reconstruction and "Image-Only" inference. Calculates bucket accuracy and latent-variable correlations.
*   **`src/visualize_attribution.py`**: The Visualizer. Generates spatial heatmaps showing which image regions correspond to specific latent dimensions (e.g., "Injury").
*   **`src/create_attribution_cohorts.py`**: The Analyst. Defines clinical cohorts (Target vs Rival) for downstream attribution analysis.

## 4. Optimal Solution
## 4. Attribution Clustering Plan

### The Recommendation: **GPyTorch Variational Shared GPLVM**
To interpret the model, we define specific clinical cohorts. These definitions guide how we analyze the latent space.

Although an Exact GP is mathematically affordable for $N=92$, a **Variational GPLVM implemented in GPyTorch** is the most robust and forward-looking solution for this repository. 

**Why?**
1.  **End-to-End PyTorch Ecosystem:** Both DINOv3 and MedSAM2 run in PyTorch. Using GPyTorch keeps the entire pipeline (from image loading to GP inference) in a single, differentiable GPU ecosystem.
2.  **Handling Missingness as a Prior:** Variational inference allows us to place a standard Normal prior over the latent space $X$. For patients with missing tabular data, the variational distribution for their latent point $q(x_i)$ will naturally rely entirely on the image data likelihood and the prior, making the cross-modal imputation statistically sound out-of-the-box.
3.  **Future-Proofing for Video Tracking:** As mentioned in `legend.txt` ("UMAP 3D spatial tracker deformation during video"), we may eventually want to embed *every frame* of an ultrasound sweep rather than a single summarized image. This will explode $N$ from 92 to tens of thousands. A Variational GP with inducing points easily scales to this regime without requiring a complete rewrite of the codebase.

1.  **DINOv3/MedSAM2 Feature Extraction:** Write a PyTorch script (using the Docker environments) to pass the `scan/` files through the encoders and extract $\mathbf{Y}^{(1)}$ (the $N \times D_1$ embedding matrix).
2.  **Dataset Curation:** Preprocess `legend.txt` targets, `PPTTUS.csv`, and `game_stats_CU.csv` to build a clean $\mathbf{Y}^{(2)}$ matrix (the $N \times D_2$ tabular matrix) representing the exact same athletes.

4.  **Training & Imputation:** Train using a variational lower bound (ELBO) and validate cross-modal imputation by artificially masking out known VISA scores/tendon thickness and predicting them from the images alone.

# Attribution Clustering Plan
## 5. How to Run

This plan details how to group the dataset into meaningful clusters. These assignments define what we are comparing when running attribution, enabling us to answer specific clinical and biomechanical questions. We will create a script `src/create_attribution_cohorts.py` that builds these definitions for the feature extractors (MedSAM2/DINOv3).

## Core Definitions

To map a clinical question into an attribution task, we need the following components for each assignment:

- **Target / Cluster A**: The group of interest (e.g., Injured Knees, Post-season scans).
- **Rival / Cluster B**: The baseline or contrasting group (e.g., Healthy Knees, Pre-season scans).
- **Centroids (`mu_pos`, `mu_neg`)**: The mathematical average embedding vector of all images/frames belonging to the Target and Rival clusters, respectively.
- **Positives**: The frames inside the Target cluster. When analyzing a specific frame, these are its nearest neighbors within the same Target cluster. They teach the model *cohesion* (what this condition generally looks like).
- **Hard Negatives**: The frames from the Rival cluster that look most structurally similar to the Target frame. They teach the model *separation* (what differentiates the target from a closely related but functionally different scan).

### Addressing the Low Sample Size Issue
*Note on Small N:* Comparing a specific player's specific knee at a single timepoint will only have ~2 still images and one DICOM video. Clustering usually requires larger distributions to find robust centroids. 
*Solution:* For questions comparing "Player X vs. Player Y", we will compute cluster centroids at the **cohort level** (e.g., all healthy vs. all injured) and compute the mathematical margin for Player X's specific frame against those global or team-level centroids. If we truly want to isolate a single knee's temporal evolution (e.g., Player X Pre vs Player X Post), we will pool individual frames extracted from the *video* scan alongside the still images to artificially inject $N$, ensuring we have enough sample points to compute a stable centroid for that specific joint at that timepoint.

## Proposed Cohort Groupings

Below are the expanded assignment groupings incorporating both the original clinical questions and the new combinatorial requirements.

### 1. Temporal Evolution (How does an image evolve over the season?)
* **Longitudinal Cohort Progression**: Target: POST (.3) L Knees. Rival: PRE (.1) L Knees. (Repeat for R knees).
* **Specific Player Progression**: Target: Player X R Knee POST. Rival: Player X R Knee PRE. Positives: Frames from Player X's POST video. Hard Negatives: Frames from Player X's PRE video. Centroids: Average embeddings of Player X POST vs Player X PRE frames.
* **Decreasing VISA-P Progression**: Target: PRE/MID/POST scans of players with dropping VISA scores.
* **Stable/Increasing VISA-P Progression**: Target: PRE/MID/POST scans of players with stable or increasing VISA scores.

### 2. Symptom Presence & Transition

* **Symptom Transition (Pre to Mid)**: 
  * Target: Non-symptomatic at PRE vs Rival: Symptomatic at MID.

Overview
We will implement a Variational Shared GPLVM using GPyTorch to map high-dimensional ultrasound feature vectors (extracted via DINOv3 or MedSAM2) and low-dimensional tabular data (
PPTTUS.csv
, game_stats) into a shared latent space. This will allow for cross-modal imputation and discovery of non-linear correlations between structural knee health and athletic performance.

Key Decisions from Discovery
Data Mapping:
Scans: T[Team]-[Player].[Timepoint]/[Knee] (e.g., T1-23.3/R).
Tabular: 
PPTTUS.csv
 contains rows mapping explicitly to the {Team}-{Player}{Knee} format (e.g., T1-23R).
Games Stats: Will need to be aggregated/mapped by Player and corresponding timepoints.
Frame Strategy (Keyframes via DPP):

Why: This ensures we capture structural variation of the tendon across the whole sweep without bloating $N$ to unreasonable sizes (e.g., hundreds of nearly identical adjacent frames per patient).

**How `train_gplvm.py` works under the hood:**

#### 1. The Inputs (`data/manifest.csv`)
The script takes the single `.csv` file we just created as its source of truth.
*   **The Split:** It first shuffles the 52 unique patients (`ppt_key` like `T1-20L`) and completely holds out 20% of the *patients* (not frames) as a validation set. This prevents the model from "cheating" by memorizing a player's knee in the training set and being tested on a different frame of that same player's knee in the validation set.
*   **The Datasets ($Y_1$ and $Y_2$):** For the remaining 80% of patients, it defines two modalities:
    *   **$Y_2$ (Tabular):** It concatenates the 17 clinical and game-stat columns (Height, Visa Score, Tendon Thickness, Minutes Played) into a single 1D vector per row. **It z-score normalizes these values** based on the training set mean/std so that variables spanning 0-100 (VISA) don't overpower variables spanning 0-1 (Thickness).
    *   **$Y_1$ (Image Embeddings):** The manifest contains the row's `frame_id` and raw image/video path.

#### 2. On-The-Fly Processing (The DataLoader)
Because DINOv3 features are massive (300MB per raw unpooled array), we do not save them. Instead, as the DataLoader builds a batch of 64 rows:
1. It reads the raw `.dcm` or `.mp4` from disk.
2. It slices out the exact `frame_id` (e.g. `f0045`).
3. It passes that single image through the frozen `DINOv3` model we just loaded locally.
4. It flattens the resulting feature tensor into a massive 1D fingerprint vector ($Y_1$).

#### 3. The Architecture (`SharedVariationalGPLVM`)
With $Y_1$ (Image) and $Y_2$ (Tabular) loaded in memory for the batch, the script pushes them into the GPyTorch model.
*   **The Latent Space ($X$):** The model initializes a small, shared 12-dimensional vector space for every patient.
*   **The Translators:** There are two separate Gaussian Process models (the "translators"):
    *   `gp_img`: Learns to map $X \rightarrow Y_1$ (768+ dimensions).
    *   `gp_tab`: Learns to map $X \rightarrow Y_2$ (17 dimensions).

#### 4. The Optimization (ELBO & Missing Data)
The model's goal is to move the $X$ coordinates around until it finds the optimal positions that can accurately generate *both* the image and the tabular data simultaneously.
*   **Variational ELBO:** It optimizes the "Evidence Lower Bound". It calculates how badly `gp_img(X)` missed the real Image ($Y_1$) and how badly `gp_tab(X)` missed the real Tabular Data ($Y_2$), sums them together, and backpropagates the error to shift $X$.
*   **The NaN Masking Magic:** If a patient is perfectly healthy but was missing their VISA survey that week, their $Y_2$ vector will have a `NaN`. The script dynamically generates a boolean mask (`mask_tab = ~torch.isnan(Y2)`) to completely zero out the loss penalty for that specific variable during that exact training step. This prevents the gradients from exploding to `NaN` and allows the model to learn the rest of the patient's data!

#### 5. The Outputs (`models/checkpoints/`)
After training, the script dumps 4 critical files into the output directory:
1.  **`shared_gplvm_model.pth`:** The core state dictionary containing the fully optimized Shared Latent Space ($X$) and the variational inducing points.
2.  **`likelihood_img.pth` & `likelihood_tab.pth`:** The optimized noise parameters for the two output spaces.
3.  **`y_stats.json`:** The Z-score normalization constants (Means and Standard Deviations) calculated on the training set. This is crucial so that when we run inference tomorrow on a *new* patient, we un-normalize their predicted VISA score using the exact same mathematics.
4.  **`optimized_latent_X.npy`:** A raw NumPy array of the final $X$ coordinates for every frame. (We can quickly load this into a UMAP clustering script to immediately visualize if the model successfully separated healthy knees from injured knees!)
